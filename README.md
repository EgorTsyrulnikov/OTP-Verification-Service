# Promo IT OTP Verification Service

Простое и удобное backend-приложение для защиты операций с помощью одноразовых кодов (OTP). Разработано для автоматической генерации, рассылки (через Email, SMS, Telegram, файл) и валидации кодов.

## Документация по каждому модулю (Структура приложения)
Приложение строго разделено на три основных слоя:
1. **Слой API (`com.promoit.otp.api`)**: Обработчики HTTP-запросов (контроллеры). Реализован с помощью **Spring MVC** (согласно допустимым альтернативам в ТЗ). Обеспечивает детальное логирование каждого запроса и обработку исключений.
2. **Слой сервисов (`com.promoit.otp.service`)**: Содержит бизнес-логику. Генерация кодов, проверка их валидности, аутентификация и фоновая задача (`ExpirationTask`), которая регулярно отмечает просроченные коды статусом `EXPIRED`. Также здесь реализованы различные каналы рассылки в подпакете `notification`.
3. **Слой DAO (`com.promoit.otp.dao`)**: Классы для выполнения запросов к БД PostgreSQL 17. Реализовано на **чистом JDBC** (`java.sql.*`), строго без использования ORM (Hibernate/JPA).

## Внешние библиотеки и их установка
Проект использует **Maven** для управления зависимостями. Все необходимые библиотеки (Spring Web, PostgreSQL JDBC, Angus Mail, JSMPP) скачиваются автоматически на этапе сборки. 

Для загрузки зависимостей и сборки проекта достаточно выполнить стандартную команду `mvn clean install`. Дополнительная ручная установка библиотек не требуется.

## Как пользоваться сервисом

Сервис работает как RESTful API. Для локального запуска:
1. Убедитесь, что запущен **PostgreSQL 17** на стандартном порту `5432`.
2. В файле конфигурации `src/main/resources/application.properties` для удобства тестирования по умолчанию прописано подключение к дефолтной базе `postgres` (пользователь `postgres`, пароль `postgres`). При необходимости данные можно легко изменить. Все таблицы (`users`, `otp_config`, `otp_codes`) создадутся автоматически при первом старте через самописный `DatabaseInitializer`.
3. Для старта приложения выполните: 
   ```bash
   mvn spring-boot:run
   ```
4. Для тестирования API можно использовать Postman, cURL или любой другой HTTP-клиент.

Настройки рассылки (SMTP, SMPP, Telegram Bot Token) вынесены в properties файлы в папке `src/main/resources/` (`email.properties`, `sms.properties`, `telegram.properties`). Для тестирования рассылки по почте достаточно указать в `email.properties` параметры вашего SMTP-сервера.

## Какие API-эндпоинты и сценарии использования поддерживаются

В системе реализовано строгое разграничение прав: администратор и обычный пользователь. Все запросы покрыты подробным логированием.

### Аутентификация (Публичный доступ)
* `POST /api/auth/register` — Регистрация нового пользователя (Ожидает JSON: `login`, `password`, `role`). Роль может быть `ADMIN` или `USER`. В системе допустим только один администратор.
* `POST /api/auth/login` — Авторизация. Возвращает JWT-подобный токен для дальнейшей работы. Токен нужно передавать в заголовке `Authorization: Bearer <token>`.

### Сценарии Администратора (Доступ только с ролью ADMIN)
* `PUT /api/admin/config` — Настройка конфигурации OTP-кодов (время жизни и количество знаков).
* `GET /api/admin/users` — Получение списка всех пользователей.
* `DELETE /api/admin/users/{id}` — Удаление пользователя (включая каскадное удаление всех его OTP-кодов).

### Сценарии Пользователя (Доступ с ролью ADMIN или USER)
* `POST /api/otp/generate` — Генерация OTP-кода.
  * *Принимает:* `operationId`, `channel` (способ отправки: FILE, EMAIL, SMS, TELEGRAM), `destination` (адрес получателя).
  * *Сценарий:* Генерирует код заданной администратором длины, отправляет его по указанному каналу и сохраняет в БД со статусом `ACTIVE`.
* `POST /api/otp/validate` — Проверка OTP-кода.
  * *Принимает:* `operationId`, `code`.
  * *Сценарий:* Проверяет правильность кода и его статус. При успешной проверке код помечается как `USED`.

## Как протестировать код

1. Запустите проект (`mvn spring-boot:run`). Сервер стартует на стандартном порту `8080`.
2. **Регистрация администратора:**
   ```bash
   curl -X POST http://localhost:8080/api/auth/register -H "Content-Type: application/json" -d "{\"login\":\"admin\",\"password\":\"1234\",\"role\":\"ADMIN\"}"
   ```
3. **Получение токена:**
   ```bash
   curl -X POST http://localhost:8080/api/auth/login -H "Content-Type: application/json" -d "{\"login\":\"admin\",\"password\":\"1234\"}"
   ```
4. **Тест генерации OTP (сохранение в файл):**
   *(Вставьте полученный токен в заголовок Authorization вместо <TOKEN>)*
   ```bash
   curl -X POST http://localhost:8080/api/otp/generate -H "Authorization: Bearer <TOKEN>" -H "Content-Type: application/json" -d "{\"operationId\":\"op_1\", \"channel\":\"FILE\", \"destination\":\"test\"}"
   ```
   *Проверка:* В корне проекта будет создан файл `otp_codes.txt` с новым сгенерированным кодом.
5. **Тест валидации OTP:**
   ```bash
   curl -X POST http://localhost:8080/api/otp/validate -H "Authorization: Bearer <TOKEN>" -H "Content-Type: application/json" -d "{\"operationId\":\"op_1\", \"code\":\"<КОД_ИЗ_ФАЙЛА>\"}"
   ```
   *Ожидаемый ответ:* `{"isValid": true}`.
6. **Тест Telegram:** В запросе генерации укажите `channel: TELEGRAM`, а в качестве `destination` — реальный `chat_id` получателя. (В `telegram.properties` уже вшит тестовый токен бота, поэтому для тестирования достаточно написать боту и узнать свой chat_id).
7. **Тест фоновой очистки:** Сгенерируйте код, подождите 5 минут (дефолтное время жизни), и убедитесь в базе данных, что его статус сменился с `ACTIVE` на `EXPIRED` (очистку производит планировщик `ExpirationTask`).
