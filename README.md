# Promo IT OTP Verification Service

Простое и удобное backend-приложение для защиты операций с помощью одноразовых кодов (OTP). Разработано для автоматической генерации, рассылки (через Email, SMS, Telegram, файл) и валидации кодов.

## Документация по каждому модулю (Структура приложения)
Приложение строго разделено на три основных слоя:
1. **Слой API (`com.promoit.otp.api`)**: Обработчики HTTP-запросов (контроллеры). Реализован с помощью **Spring MVC** (одобрено в ТЗ как альтернатива `com.sun.net.httpserver`). Обеспечивает детальное логирование каждого запроса и обработку ошибок.
2. **Слой сервисов (`com.promoit.otp.service`)**: Содержит бизнес-логику. Генерация кодов, проверка их валидности, аутентификация и фоновая задача (`ExpirationTask`), которая регулярно отмечает просроченные коды статусом `EXPIRED`. Также здесь находятся сервисы рассылки в подпакете `notification`.
3. **Слой DAO (`com.promoit.otp.dao`)**: Классы для выполнения запросов к БД PostgreSQL 17. Реализовано на **чистом JDBC** (`java.sql.*`), без использования ORM-библиотек.

## Внешние библиотеки и их установка
В качестве системы сборки используется **Maven**. Все внешние библиотеки скачиваются автоматически. 
* `spring-boot-starter-web` — для API (Spring MVC).
* `spring-boot-starter-jdbc` и `postgresql` — для работы с БД.
* `angus-mail` (Jakarta Mail) — для отправки Email.
* `jsmpp` — для отправки SMS по протоколу SMPP.

**Инструкция по установке библиотек:** Вам не нужно ничего скачивать вручную. Достаточно выполнить команду `mvn clean install`, и Maven сам подтянет все нужные библиотеки.

## Как пользоваться сервисом

Сервис работает как RESTful API. Для использования сервиса необходимо:
1. Иметь установленную СУБД **PostgreSQL 17**.
2. В файле `src/main/resources/application.properties` указана база данных `postgres` (стандартная БД, создающаяся при установке PostgreSQL). Таблицы (`users`, `otp_config`, `otp_codes`) создадутся автоматически при старте приложения!
3. Запустить приложение с помощью Maven: `mvn spring-boot:run`
4. Использовать HTTP-клиент (например, Postman или cURL) для отправки запросов к API.

Настройки рассылки (SMTP, SMPP, Telegram Bot Token) лежат в папке `src/main/resources/` (`email.properties`, `sms.properties`, `telegram.properties`).

## Какие API-эндпоинты и сценарии использования поддерживаются

Все запросы логируются. В системе реализовано строгое разграничение прав: администратор и пользователь.

### Аутентификация (Публичный доступ)
* `POST /api/auth/register` — Регистрация нового пользователя (Ожидает JSON: `login`, `password`, `role`). Роль может быть `ADMIN` или `USER`. В системе может быть только один администратор.
* `POST /api/auth/login` — Логин. Возвращает JWT-подобный токен для дальнейшей работы. Токен нужно передавать в заголовке: `Authorization: Bearer <token>`.

### Сценарии Администратора (Только роль ADMIN)
* `PUT /api/admin/config` — Изменение конфигурации OTP-кодов (время жизни и количество знаков).
* `GET /api/admin/users` — Получение списка всех простых пользователей.
* `DELETE /api/admin/users/{id}` — Удаление пользователя и каскадное удаление всех его OTP-кодов.

### Сценарии Пользователя (Роль ADMIN или USER)
* `POST /api/otp/generate` — Генерация OTP-кода, привязанного к операции.
  * *Принимает:* `operationId`, `channel` (способ отправки: FILE, EMAIL, SMS, TELEGRAM), `destination` (адрес получателя).
  * *Сценарий:* Генерирует код заданной администратором длины, отправляет его по указанному каналу и сохраняет в БД со статусом `ACTIVE`.
* `POST /api/otp/validate` — Валидация OTP-кода, высланного пользователю.
  * *Принимает:* `operationId`, `code`.
  * *Сценарий:* Проверяет правильность кода и его статус. Если всё верно, код помечается как `USED`.

## Как протестировать ваш код

1. Запустите проект (`mvn spring-boot:run`). По умолчанию сервер стартует на порту `8080`.
2. **Регистрация администратора:**
   ```bash
   curl -X POST http://localhost:8080/api/auth/register -H "Content-Type: application/json" -d "{\"login\":\"admin\",\"password\":\"1234\",\"role\":\"ADMIN\"}"
   ```
3. **Получение токена:**
   ```bash
   curl -X POST http://localhost:8080/api/auth/login -H "Content-Type: application/json" -d "{\"login\":\"admin\",\"password\":\"1234\"}"
   ```
4. **Тест генерации OTP (сохранение в файл):**
   *(Вставьте полученный токен в заголовок Authorization)*
   ```bash
   curl -X POST http://localhost:8080/api/otp/generate -H "Authorization: Bearer ВАШ_ТОКЕН" -H "Content-Type: application/json" -d "{\"operationId\":\"op_1\", \"channel\":\"FILE\", \"destination\":\"test\"}"
   ```
   *Проверка:* Откройте файл `otp_codes.txt` в корне проекта. В нем должен появиться свежий сгенерированный код.
5. **Тест валидации OTP:**
   ```bash
   curl -X POST http://localhost:8080/api/otp/validate -H "Authorization: Bearer ВАШ_ТОКЕН" -H "Content-Type: application/json" -d "{\"operationId\":\"op_1\", \"code\":\"КОД_ИЗ_ФАЙЛА\"}"
   ```
   *Ожидаемый ответ:* `{"isValid": true}`.
6. **Тест Telegram:** В запросе `/api/otp/generate` укажите `channel: TELEGRAM`, а в качестве `destination` — ваш реальный `chat_id`. (Бот предварительно должен быть запущен в Telegram, а токен указан в `telegram.properties`).
7. **Тест фоновой очистки:** Сгенерируйте код, подождите 5 минут (дефолтное время жизни), и убедитесь в базе данных, что его статус изменился с `ACTIVE` на `EXPIRED` (задачу выполняет `ExpirationTask`).
